Improving Netezza Performance: Best Practices

1) Considerations for better performance:

Distribution

Bad distribution can cause data skew (more data stored in certain data slices while less data is stored in other data slices). This affects query performance as the data slice with more work to do impacts the entire system. If you're unsure of the right distribution key, consider distributing on random.

Data types matter too. Choose a type that minimizes disk storage requirements and scan time. Using Char(x) instead of Varchar(x) for fixed length data can save space and reduce I/O.

Statistics

The nzsql GENERATE STATISTICS command gathers statistics about each table column‚Äôs proportion of stats, including duplicate values, maximum value, minimum value, null values, dispersion values and updates the system catalog table. The optimizer relies on these statistics to determine the most efficient way to execute a query.

Zone maps

Zone maps are used when you have numeric datatypes for join columns and where clause columns. They help optimize the processing of queries.

Groom table command

Groom tables that receive frequent updates or deletes, or if a load or insert is aborted. Grooming at a record level can remove all deleted records regardless of their location.

2) Performance Issues:

Common reasons for performance issues

  • Inappropriate distribution keys can cause data skew.
  • Process skew can occur when not taking into account processing patterns, even with uniform data distribution.
  • Large volume data re-distribution or broadcast during query execution impacts performance.
  • Consider numeric datatypes for join columns and where clause columns to enable zone maps usage.
  • Table data that isn't organized optimally for multi table joins in a data warehouse environment affects performance.

Steps to do query performance analysis

  1. Identify long running queries or those queuing other queries.
  2. Generate query plans using the ‚ÄúEXPLAIN‚Äù command or analyze .pln files in the data.<ver>/plans directory under the /nz/data directory.
  3. Look for performance issues details from the previous sections and take necessary actions like generating statistics, changing distributions, grooming tables, etc.
  4. Verify whether the modifications helped with the query plan by regenerating the plan using the explain utility.

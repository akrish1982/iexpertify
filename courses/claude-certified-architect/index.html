Domain 1 Notes: Agentic Architecture & Orchestration

The Agentic Loop

  • Core cycle: Send request -> inspect stop_reason -> execute tools or terminate.
  • stop_reason: "tool_use" means Claude wants tools. Continue the loop.
  • stop_reason: "end_turn" means Claude is finished. Terminate the loop.
  • Tool results must be appended to conversation history before the next iteration or Claude cannot reason over the tool output.
  • stop_reason is the reliable termination signal. It is deterministic and unambiguous.

Three Anti-Patterns

Anti-pattern Why it fails
Parsing natural language like "I'm done" Ambiguous. Claude may be done with one step while still intending to continue the task.
Iteration caps as the primary loop control Either cuts off useful work too early or wastes cycles on unnecessary turns.
Checking only content[0].type == "text" Claude can return text and tool_use blocks in the same response.

Use iteration caps only as a safety net to prevent runaway agents, never as the primary stopping mechanism.

Orchestration Patterns

Pattern When to use Key characteristic
Sequential Each step depends on the previous result A -> B -> C, with state flowing from one step to the next
Parallel Subtasks are independent and latency matters Fan-out, fan-in, with no shared state between branches
Pipeline Stages have different specializations Assembly line where one stage feeds the next
Dynamic adaptive Task structure is unknown upfront The model decides decomposition at runtime
Hub-and-spoke A coordinator delegates to specialists Central orchestration with focused subagents

Decision rule: Use parallel when subtasks are independent. Use sequential when each step needs the previous output. Use dynamic adaptive when you cannot predict the decomposition at design time.

Guardrails Hierarchy

Prompt instructions are probabilistic. Hooks are deterministic. That distinction is one of the most exam-tested ideas in this domain.

Mechanism Type Enforcement Use for
System prompt rules Probabilistic Model may not comply Style guidance and soft preferences
PreToolUse hooks Deterministic Code-level, before execution Blocking dangerous calls and validating parameters
PostToolUse hooks Deterministic Code-level, after execution Validating outputs, sanitizing results, and audit logging

Hook order matters: PreToolUse fires before the tool runs. PostToolUse fires after. When the question says "must always" or "guaranteed", the answer is hooks, not prompts.

Claude Agent SDK and Multi-Agent Systems

  • AgentDefinition declares identity, system prompt, and available tools.
  • allowedTools should scope each agent to about 4-5 tools.
  • The task tool lets a coordinator delegate work to a subagent that runs in its own context.
  • Handoffs transfer control between agents, and the receiving agent does not inherit conversation history.
  • Subagents do not share memory. All context must be passed explicitly in task definitions or handoff payloads.
  • Specialists improve accuracy because they have smaller toolsets, smaller context windows, and clearer responsibilities.

Human-in-the-Loop and Recovery

Never silently fail. If the agent cannot complete the task, emit a structured handoff instead of a generic error.

Customer ID: 12847
Summary: Tool calls completed, but the policy exception requires human approval.
Root cause: Confidence fell below threshold after conflicting eligibility checks.
Recommended action: Review the exception and approve or deny the operation.

Escalate when confidence is below threshold, policy exceptions apply, destructive operations are requested, or intent stays ambiguous after available context is exhausted.

Error Recovery & Resilience

Strategy When to use
fork_session Divergent exploration where you want to try alternatives without polluting the main context
Fresh start + summary injection Context is stale, contradictory, or polluted
Retry with error feedback Transient failure where the model can recover if you show the error explicitly
Graceful degradation Partial results are still useful and better than a hard failure

Stale context signals include repetition, contradiction, or ignoring recent tool results. The fix is a fresh start with a curated summary, not more context piled into the thread.

Domain 2 Notes: Tool Design & MCP Integration

Tool Description Design

Tool descriptions matter more than tool names because they are the primary selection signal Claude uses when deciding what to call.

  • State what the tool does in one sentence.
  • Document expected input formats and constraints.
  • Describe the response shape Claude will get back.
  • State the boundary conditions, including what the tool does not do.
  • Include example queries that should trigger the tool.

Optimization sequence: Improve descriptions first, then add a few-shot example, then consolidate tools if needed. Do not start by reducing tool count.

Schema Design Rules

  • Keep agents to about 4-5 tools. More tools increase selection errors.
  • Use descriptive parameter names like customer_email instead of email.
  • Mark parameters as required only when truly mandatory.
  • Use enums for constrained choices because they guide the model better than freeform strings.
  • If Claude misselects a tool, improve the description before consolidating tools.

tool_choice Modes

Mode Behavior Use when
auto Model decides whether to call a tool Default for most agentic loops
any Model must call at least one tool, but chooses which You know a tool is required but do not care which one
tool (forced) Model must call a specific named tool You need guaranteed structured output or strict schema compliance

Exam shortcut: tool_choice: { type: "tool", name: "extract_data" } is the right answer when the prompt says "guaranteed structured output."

MCP Architecture

Layer Role Example
Client Connects to servers and routes tool calls Claude Desktop or an IDE extension
Host Manages client lifecycle The desktop app process
Server Exposes tools, resources, and prompts Database, filesystem, or API connector

Protocol: JSON-RPC 2.0 over stdio or HTTP/SSE. Project configuration belongs in .mcp.json. Personal global configuration belongs in ~/.claude.json. Use community MCP servers first and build custom ones only when necessary.

Structured Tool Errors

{
  "errorCategory": "authentication",
  "isRetryable": false,
  "retryAfterMs": 5000,
  "partialResult": null,
  "suggestion": "Check API key permissions"
}
  • Access failure like 401 or timeout means retry or escalate.
  • Valid empty result like 0 search matches is a successful outcome, not an error.
  • Never return a generic error string when structured metadata can support recovery.

Tool Selection in Claude Code

Tool Purpose Use when
Grep Search file contents by pattern You are looking for code patterns or string occurrences
Glob Find files by name or path pattern You need discovery by extension or naming convention
Read Read a specific file You already know the exact path
Edit Modify file contents You are making targeted changes to existing files
Bash Run shell commands The task is build, test, git, or otherwise outside specialized tools

Selection principle: Use the most specific tool available. Grep for content search, Glob for discovery, and Read for known files. Avoid Bash when a more precise tool exists.

Domain 3 Notes: Claude Code Configuration & Workflows

Configuration Hierarchy

Priority Location Scope Committed to Git?
1 ~/.claude/CLAUDE.md User-global No
2 .claude/CLAUDE.md Project-wide Yes
3 CLAUDE.md in any directory That directory and below Yes
4 .claude/rules/*.md Conditional, path-matched Yes

More specific scope wins. A directory-level CLAUDE.md overrides the project-level file for paths in that subtree.

Path-Scoped Rules

---
paths:
  - "src/api/**"
  - "src/middleware/**"
---
Always validate authentication tokens before processing API requests.
Use structured error responses with proper HTTP status codes.

Rules in .claude/rules/ load only when the active files match the declared path globs, which avoids wasting context on irrelevant instructions.

Skills and Commands

Capability Location What it is for
Project skills .claude/skills/ Shared reusable workflows with restricted tool access
Personal skills ~/.claude/skills/ Personal reusable workflows
Project commands .claude/commands/ Shared prompt templates invoked with /
Personal commands ~/.claude/commands/ Personal prompt templates
  • Skills are capability modules and can restrict tools with allowed-tools.
  • context: fork lets a skill run in a forked context so it does not pollute the main conversation.
  • Commands are prompt templates that may include $ARGUMENTS.
  • Skills and commands are not the same thing.

Hooks and Working Modes

Concept Correct use
PreToolUse Block dangerous calls, validate parameters, and require confirmation before execution
PostToolUse Validate outputs, sanitize results, and write audit logs after execution
Plan mode Use for complex work with multiple reasonable approaches
Direct execution Use for clear, well-defined tasks with obvious execution steps
-p flag Use for CI/CD and non-interactive automation, not for plan mode

When the question says "cannot be bypassed", the answer is a hook. When it says "preferred style" or "guidance", the answer is prompt instructions in CLAUDE.md.

Feedback, Review, and Permissions

  • Concrete examples are better than prose-only guidance. Show the code you want.
  • Batch independent fixes in one message. Sequence dependent changes across messages.
  • Never review code in the same session that wrote it. Use an independent session to remove reasoning bias.
  • Permissions are granted at the tool level. Keep every agent or skill on least privilege.
  • Project settings in .claude/ are shared. Personal settings in ~/.claude/ are not.

Domain 4 Notes: Prompt Engineering & Structured Output

System Prompts

  • Structure prompts with role and context, rules and constraints, output format, and calibration examples.
  • Concrete code examples beat prose descriptions because they show the expected behavior directly.
  • Severity calibration matters. Without examples, the model tends to flatten critical issues and small nits into the same priority.
  • System prompts are cost-effective because prompt caching reuses them across requests.

Structured Output: Forced Tool Use vs Text

Method Guarantee Use when
Forced tool_choice Schema enforced by the API You need guaranteed structure every time
tool_choice: auto No guarantee that a tool will be used Agentic loops where text responses are also valid
Prompt-based JSON No schema enforcement Simple prototypes only

The preferred production pattern is to define a dedicated output tool such as extract_entities and force it. That gives API-level schema validation and avoids malformed JSON.

Schema Design for Structured Output

  • Make fields optional or nullable if the value may truly be missing.
  • Required fields can force the model to fabricate data when the source does not contain a value.
  • Use enums for constrained fields.
  • Keep schemas flat where possible. Deep nesting increases extraction errors.
  • Add descriptions on each property so the model knows exactly what belongs there.

Few-Shot Examples and Prompt Chaining

  • Use few-shot examples when instructions alone produce inconsistent output.
  • 2-4 examples is the sweet spot. More rarely helps enough to justify the extra context.
  • Include reasoning and edge cases, not just input-output pairs.
  • Use prompt chaining when the task has distinct phases such as extract, validate, format, and synthesize.
  • Chaining improves accuracy because each step has a smaller focused prompt and can be retried independently.

Retry With Error Feedback

Original prompt: "Extract all dates from this contract"
Failed output: { "dates": ["2024-01-15", "next Tuesday"] }
Validation error: "dates[1] is relative. All dates must be ISO 8601."

Never say only "try again." Feed back the original request, the failed output, and the exact validation error so the model can correct itself precisely.

Batch API and Review Limits

Concept Correct interpretation
Batch API Use for bulk, latency-tolerant workloads because it is cheaper, not faster
Real-time user-facing work Use synchronous APIs, not Batch
Self-review Use a separate model instance or fresh conversation, not the same session
Large input review Use per-file passes and then a cross-file integration pass

Domain 5 Notes: Context Management & Reliability

Persistent Facts vs Progressive Summaries

Do not rely on repeated summarization to preserve important details. Each summary pass drops names, numbers, timestamps, and exact wording. Instead, keep a structured fact block and carry it forward verbatim.

## Persistent Facts
- Customer: Jane Smith (ID: 12847)
- Order: #ORD-2024-8821, placed 2024-01-15
- Issue: Delivery address incorrect
- Status: Refund approved, reshipment pending

Scratchpad Files and Context Resets

  • Use scratchpad files to preserve findings across long-running explorations.
  • Conversation history is internal persistence and can be lost when context becomes stale.
  • Scratchpad files are external persistence and survive fresh starts.
  • Use fresh start + summary injection when the model repeats itself, contradicts itself, or ignores recent information.

Structured Error Propagation

Field Purpose Example
Failure type Describe the category of failure network_timeout
Partial results Show what succeeded { "orders": [...], "payments": null }
Alternatives tried Record recovery attempts Retried 2x, tried fallback endpoint
Suggested action Guide the next agent or human Escalate to support with order IDs

Generic errors like "something went wrong" are not useful. They block recovery because the next step has no actionable context.

Valid Empty Results vs Access Failures

Situation Type Correct action
Search returns 0 results Valid empty Accept it. The absence of data is the answer.
Database query returns an empty set Valid empty Accept it. The record does not exist.
API returns 401 Access failure Retry with correct credentials or escalate.
Network timeout Access failure Retry or use a fallback path.
Rate limit 429 Access failure Respect retry timing and try again.

Validation, Attribution, and Monitoring

  • High overall accuracy can hide weak categories. Use stratified validation and per-type accuracy checks.
  • When users report errors but aggregate metrics look good, break results down by type, source, or category.
  • Use structured claim-source mappings instead of inline citations if the content will pass through multi-step synthesis.
  • If sources conflict, annotate both claims with provenance and dates. Do not silently average or overwrite one with the other.
  • Track per-type error rates, confidence calibration, retry success rates, and context window utilization.

Context Window Strategies

Strategy When to use
Persistent fact blocks Critical details must survive long conversations
Scratchpad files Multi-step exploration crosses context boundaries
Per-file passes + integration Large codebases risk attention dilution
Fresh start + summary injection Context has gone stale or contradictory
Prompt caching Repeated system prompts need lower cost and latency

Self-Check

Which mechanism guarantees a dangerous tool call is blocked every time?

PreToolUse hooks are deterministic code-level enforcement. Prompts are probabilistic and iteration caps solve runaway loops, not policy enforcement.

Try It Snippet

while (true) {
  const response = await client.messages.create(request);

  if (response.stop_reason === "tool_use") {
    const toolResults = await runRequestedTools(response);
    history.push(response);
    history.push(toolResults);
    continue;
  }

  if (response.stop_reason === "end_turn") {
    history.push(response);
    break;
  }

  throw new Error("Unexpected stop_reason: " + response.stop_reason);
}
